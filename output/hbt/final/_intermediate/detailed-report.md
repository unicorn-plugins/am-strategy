## 2.1 비즈니스 동인 매핑 — 4S + 혁신 5개 전수 검사

### 2.1.1 5개 동인 개요

| # | 동인 | HBT 현황 | DORA 2025 포지션 | AM 필요성 |
|---|------|---------|-----------------|---------|
| 1 | **Speedy** | 신규 기능 Lead Time 평균 **22주** | 글로벌 하위 2% (98%가 1개월 이내) | **필수** — 경쟁사 대비 5~10배 느림 |
| 2 | **Service Always** | 장애 복구 평균 **4.8시간**, CFR **24%** | 하위 40%, CFR 상위 62% 구간 | **필수** — SLA 위반 연 12건 |
| 3 | **Save Cost** | IT 운영비 증가율 **+7.3%** vs 매출 **+1.8%** | 4배 격차 지속 시 3년 후 영업이익률 -2.1%p | **필수** — 비용 구조 붕괴 임박 |
| 4 | **Security** | 레거시 CVE 미패치 **137건**, SAST 자동화 **0%** | 금감원 2026 제재 대상 후보 | **필수** — 규제 리스크 정량 |
| 5 | **혁신(AI)** | AI Copilot 파일럿 ROI 측정 불가 (2025) | Figure 49: 플랫폼 품질 없이 ROI ≈ 0 | **필수** — AI 투자 자체가 좌초 |

> **결론**: 5개 동인 모두 "AM을 해야 한다/안 해야 한다"가 아니라 **"얼마나 빨리 해야 하는가"의 문제**.

### 2.1.2 동인 1 상세: Speedy — 속도 위기

HBT의 신상품 출시 Lead Time은 **평균 22주(154일)**로, 글로벌 DORA 2025 조사 대비 하위 2%에 해당함.  
업계 평균(경쟁사 SKT·KT 기준) 6~8주와 비교하면 **3~4배** 느린 수준.

**원인 분석** (Step 2 인벤토리 기반):

- BILL-MF 단일 배포 의존: 요금 관련 기능 변경 시 BILL 전체 릴리스 필요, 평균 12주
- CRM 데이터 의존성 높음(11개 연결): 한 번에 배포 범위가 지나치게 넓음
- 테스트 자동화율 거의 0%: 회귀 테스트 수동, QA 4주 소요
- 배포 프로세스 표준화 미흡: 환경별 수작업 배포, 평균 3주

**목표**: 3년 내 Lead Time → **1~7일** (22주 대비 95~99% 단축, 근로복지공단 23배 단축 사례 근거)

### 2.1.3 동인 2 상세: Service Always — 안정성 위기

**현황**:
- 장애 복구 시간 (MTTR): 평균 **4.8시간** (DORA Top 21.3% 기준 1시간 이내 대비 5배 초과)
- Change Fail Rate (CFR): **24%** (배포 4건 중 1건이 장애 유발, DORA Top 36.2% 기준 8% 대비 3배)
- SLA 위반: 연 12건, 위약금 연 **430~1,140억** (취약 운영 시스템 기준 최대치)

**원인**: 단일 배포 단위 과도하게 큼 → 배포 리스크 집중 / 롤백 자동화 없음 / IDP SPOF

### 2.1.4 동인 3 상세: Save Cost — 비용 구조 위기

**비용 격차 심화**:
- IT 운영비 증가율: +7.3%/년
- 매출 증가율: +1.8%/년
- **격차**: 4배 → 현 추세 지속 시 3년 후 영업이익률 -2.1%p 하락 예측

**기술부채 실질 비용 (연 890~1,600억)**:

| 부채 항목 | Conservative | Base | Optimistic |
|---------|------------|------|-----------|
| 레거시 유지보수 오버헤드 | 180억 | 220억 | 260억 |
| 장애·SLA 위약금 | 120억 | 160억 | 200억 |
| 보안 취약점 대응 | 60억 | 100억 | 140억 |
| 인재 이탈·채용 | 80억 | 115억 | 150억 |
| 신규 기능 기회비용 | 250억 | 350억 | 450억 |
| **AI ROI 잠금 기회비용** | **200억** | **300억** | **400억** |
| **합계** | **890억** | **1,245억** | **1,600억** |

> 운영비 620억 대비 기술부채 비용이 **2.0~2.6배** — "운영비가 작아도 기술부채가 더 큰 실제 비용"

### 2.1.5 동인 4 상세: Security — 보안·규제 위기

- 레거시 CVE 미패치 건수: **137건** (CVSS 7.0 이상 고위험 포함)
- SAST 자동화율: **0%** (전수 수동 취약점 점검)
- 금감원 2026 전자금융감독규정 강화 대상 후보 — 통신사 결제·빌링 시스템 포함
- 보안 대응 연간 비용: 60~140억 (인건비·외주·대응 비용)

**목표**: Phase 2 완료 시 SAST 자동화 100%, CVE 미패치 **0건** 유지

### 2.1.6 동인 5 상세: 혁신(AI) — AI ROI 잠금 위기

**DORA 2025 Figure 49 핵심 인사이트**:

> *"An investment in AI without a corresponding investment in high-quality platforms  
> is unlikely to yield significant returns at the organizational level."*  
> — DORA 2025 Platform Engineering Chapter

**HBT 2025 AI Copilot 파일럿 결과**:
- 개발 속도 향상: 측정 가능 (개인 단위)
- 조직 단위 ROI: **측정 불가** — 레거시 통합 복잡도로 상쇄
- 결론: DORA Figure 49 예측과 정확히 일치

**AM이 AI ROI의 전제조건인 이유**:
- AI가 생성한 코드를 Small Batch로 배포할 인프라 없음 → Rework Rate 증가
- 테스트 자동화 없이 AI 코드 품질 보증 불가 → CFR 악화
- 레거시 데이터 사일로 → AI 컨텍스트 구성 불가 → RAG/MCP 활용 불가

## 2.5 "시간 할인(Time Discount)" — 연기의 역설

AM을 1년 늦출 때마다:
- 기술부채 연 +10~15% 누증 (AI 시대에는 +20%) → **1년 지연 = 178~240억 손실 (Base)**

> **"연기 비용 > 투자 비용"이 2026년부터 영구 성립.**

- 기술 문서 초안 작성
- 아키텍처 다이어그램 생성 지원
- 버그 분석 (비고객 데이터)
- 코드 리팩터링 제안
- API 문서 생성
- 학습용 튜토리얼 생성
- 공개 벤치마크 분석
- 일정 관리 보조
- 비독점 데이터 기반 분석
- 내부 Q&A 시스템 (비PII 데이터)

**Permitted with Guardrails (조건부 허용, 8개 use case)**:

- 사내 AI 도구로만 독점 소스 코드 사용 (외부 AI 불가)
- 모든 AI 생성 코드 human-in-the-loop 필수
- PII 마스킹 후 고객 데이터 패턴 분석
- 내부 RAG 시스템으로 사내 문서 검색
- AI 기반 코드 자동완성 (PR 라인 수 <300 전제)
- 네트워크 로그 AI 분석 (내부 클러스터 한정)
- 암호화된 고객 정보 내부 AI로 분석
- AI 기반 장애 예측 (내부 데이터만)

**Prohibited (금지, 10개 use case)**:

- 고객 PII/영업비밀을 공개 AI 모델 입력
- 보안 키/비밀번호 프롬프트 포함
- 규제 데이터 (금융정보 등) 외부 전송
- KT 표준 프로토콜 변경 관련 AI 생성 코드 직접 배포
- COBOL 레거시 자동 변환 코드 검증 없이 프로덕션 반영
- 고객 통화 내용 외부 AI 전송
- 과금·빌링 엔진 로직 AI 자동 생성 후 미검증 적용
- 개인식별 네트워크 트래픽 데이터 외부 모델 학습
- 경쟁사 비교 프롬프트에 내부 KPI 포함
- AI 생성 인프라 코드 (IaC) 보안 검토 없이 배포

## 5.3 부록 C: 아키텍처 패턴 상세

### C.1 Strangler Fig 패턴 (BILL-MF 적용)

```
[BILL-MF 레거시]           [BILL-MF 신규 마이크로서비스]
   COBOL/Oracle    ←─────   요금엔진 BC (새 서비스)
        │                   상품 BC (새 서비스)
        │   Facade Layer    정산 BC (새 서비스)
        └──→ [API Gateway] ──→ [점진적 트래픽 이관]
                               0% → 10% → 50% → 100%
```

**적용 순서**:
1. 저위험 도메인 분리: 상품 정산 (KT 비의존)
2. 중위험 도메인 분리: 대리점 정산
3. 고위험 도메인 분리: 요금 엔진 (KT 기술 합의 후)
4. 레거시 COBOL 정지 (데이터 마이그레이션 완료 후)

### C.2 Blue-Green 배포 (파일럿 DLR-PORTAL 적용)

```
현재 환경 (Blue)  →  트래픽 100%
신규 환경 (Green) →  트래픽 0% (준비)
배포 후:
현재 환경 (Blue)  →  트래픽 0% (대기, 롤백 대기)
신규 환경 (Green) →  트래픽 100%
문제 발생 시:
현재 환경 (Blue)  →  트래픽 100% (5분 내 롤백)
신규 환경 (Green) →  트래픽 0%
```
