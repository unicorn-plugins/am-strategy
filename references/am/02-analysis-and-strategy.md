# 02. AM 현황분석 및 전략 수립

> **커리큘럼 목표**  
> AM 전환 여정 5단계(WHY 정의 → 현황분석 → 전략수립 → 실행 → 운영최적화)의 각 단계별 산출물과 방법론을 학습함.  
> 특히 현황분석(3단계)과 전략수립(5영역) 실무에 바로 적용 가능한 가이드 제공.

---

## 2.1 AM 전환 여정 개요

| STEP | 단계 | 핵심 활동 |
|------|------|---------|
| 1 | WHY 정의 | 비즈니스 동인 정의(4S), 기대 성과 정량화, 경영진 스폰서십 확보 |
| 2 | 현황분석(As-Is) | ① 시스템 인벤토리 구축 ② 건강도 평가 ③ 전환 적합도 판정 |
| 3 | 전략수립(To-Be) | ① 6R 전략 상세화 ② TCO 분석 ③ 리스크 평가 ④ 거버넌스/가드레일 ⑤ 변화관리 기획 |
| 4 | 실행(전환) | 파일럿 실행, 점진적 전환, 데이터 분리, 배포 자동화, 복원력 테스트, 변화관리 실행 |
| 5 | 운영최적화 | 성능 튜닝, FinOps, 통합 관제, 레거시 폐기, 거버넌스 유지 |

## 2.2 Step 1 — WHY 정의

### 비즈니스 동인 정의 (4S 매핑)

| 동인 | 4S 매핑 | 기대 성과 예시 |
|------|---------|---------------|
| 민첩성 확보 | Speedy | 배포주기 3일 → 1일, 시장 대응 3배 향상 |
| 서비스 안정성 | Service Always | SLA 99.9% → 99.99%, MTTR 4h → 15min |
| 비용 최적화 | Save Cost | 인프라 비용 30%↓, 라이선스 50%↓ |
| 보안 강화 | Secirotu | EOL S/W 제거, 제로트러스트 적용 |
| 혁신 역량 | — | AI/ML·클라우드 신기술 통합 환경 확보 |

### 경영진 스폰서십 확보 — 3단계

| Level | 시점 | 방법 |
|-------|------|------|
| L1 | WHY 정의 | 벤치마크로 방향성 제시 → "업계 상위 25% 수준 목표" |
| L2 | 현황분석 | TCO 분석으로 비용 근거 → "안 하면 매년 Y억 누적" |
| L3 | 파일럿 후 | 실증 데이터로 확정 승인 → "실증 완료, 본격 전환 GO" |

> **KPI 합의**: 경영진과 성과 지표를 사전에 합의.

### 단계적 정량화 (Progressive Quantification)

처음부터 정확한 숫자는 불가능 — 3단계로 접근.

**L1 — 업계 벤치마크 활용 (DORA Metrics)**

DORA(DevOps Research and Assessment, Google 산하)의 4대 핵심 성과 지표 — *State of DevOps Report* 연간 발표.


| 지표 | 업계 하위 | 업계 상위 | 목표 범위(상위 25% 기준) |
|------|----------|----------|------------------------|
| 배포 빈도 | 월 1회 | 일 수회 | 월 → 주 단위 개선 |
| 리드타임 | 1~6개월 | 1시간 이내 | 주 → 일 단위 |
| MTTR | 1주 이상 | 1시간 이내 | 현재 대비 50%↓ |
| 변경 실패율 | 46~60% | 0~15% | 30% 이하 |

> 단일 숫자가 아닌 **범위**(보수적 ~ 낙관적)로 제시 권장: "인프라 비용 20~40% 절감".

**L2 — 자동 도구로 베이스라인 확보**

| 측정 대상 | 자동 수집 도구 | 실무진 역할 |
|----------|--------------|------------|
| 코드 품질·기술 부채 | CAST Highlight / SonarQube | 코드 저장소 접근 권한 제공 |
| 응답시간·장애 빈도 | APM (Prometheus 등) | 모니터링 데이터 공유 |
| 배포 주기 | CI/CD 로그 | 파이프라인 로그 공유 |
| 인프라 비용 | 클라우드 비용 대시보드 | 재무팀 제공 |

**L3 — 파일럿 실측 데이터로 확정**

- 파일럿(1~2개월) 후 실제 달성 데이터 수집 → 약속이 아닌 **증거 기반 정량화**
- 예: "파일럿에서 배포 주기 5일 → 0.5일 실증 → 전체 적용 시 연 X억 절감"

> L1(벤치마크) → L2(자동 측정) → L3(파일럿) 순서로 부담 없이 정량화.

## 2.3 Step 2 — 현황분석

### ① 시스템 인벤토리 구축

**수집 항목**

| 카테고리 | 수집 항목 | 수집 방법 |
|---------|---------|---------|
| 기본 정보 | 애플리케이션명, 소유팀, 사용자 수, 매출 기여도 | 비즈니스팀 인터뷰, CMDB |
| 기술 스택 | 언어, 프레임워크, DB, 미들웨어, OS | 개발팀 조사 |
| 아키텍처 | 모놀리스/MSA, 배포 방식, 인프라(온프레미스/클라우드) | EA팀, 운영팀 |
| 운영 현황 | 장애 빈도, MTTR, SLA, 배포 주기 | 모니터링 시스템, 인시던트 로그 |
| 비용 | 인프라·라이선스·인건비 | 재무팀, 계약서 |
| 의존성 | 타 시스템 연동 포인트, 공유 DB | 개발팀, 네트워크 맵 |

**A/B/C 등급 분류**
- **A**: 핵심 비즈니스(주문·결제·고객관리) → 전환 시 최대 주의
- **B**: 지원 업무(리포팅·알림·내부도구) → 파일럿 후보
- **C**: 사용 빈도 낮음 → Retire/Replace 후보

> **변화관리 조기 착수**: 이 단계부터 시작. 이해관계자 맵 작성, 참여형 워크숍으로  
> "우리 시스템의 현황을 함께 진단" — AM을 '당하는 것'이 아닌 '함께하는 것'으로 인식.

### ② 건강도 평가 — 4차원 스코어카드

| 차원 | 평가 항목 | 1점(위험) | 5점(양호) |
|------|----------|----------|----------|
| 비즈니스 가치 | 매출 기여도, 사용자 수, 전략 정렬도 | 사용자 거의 없음 | 핵심 수익원, 다수 활성 사용자 |
| 기술 품질 | EOL 현황, 테스트 커버리지, 배포 주기 | EOL 경과, 테스트 0% | 최신 LTS, 커버리지 80%+ |
| 데이터 결합도 | 공유 DB 수, 서비스 간 JOIN, 트랜잭션 | 5개+ 서비스가 단일 DB 공유 | 독립 DB, API로만 접근 |
| 운영 안정성 | 장애 빈도, MTTR, SLA 달성률 | 월 3회+ 장애, MTTR 4시간+ | 무장애, MTTR 15분 이내 |

### 기술 부채 비용 산정

"전환하지 않았을 때의 유지 비용 리스크"를 정량화.

| 항목 | 산정 방법 | 예시 |
|------|---------|------|
| 유지보수 추가 공수 | 레거시 수정 시 추가 소요시간 × 인건비 | 동일 기능에 3배 소요 → 연 2억 추가 |
| 장애 대응 비용 | 장애 횟수 × MTTR × 투입 인력 × 인건비 | 월 3회 × 4시간 × 5명 |
| 보안 취약점 리스크 | EOL SW 보안사고 피해액 × 발생 확률 | 데이터 유출 10억 × 5% = 연 5천만 |
| 인재 이탈/채용 비용 | 레거시 기피로 인한 채용난·이직률 증가분 | 채용 기간 2배, 연봉 프리미엄 15% |
| 기능 지연 기회 비용 | 신기능 출시 지연 → 시장 선점 실패 매출 | 경쟁사 대비 3개월 지연 → 분기 매출 5%↓ |

**평가 방법 가이드**
- **자동 분석**: CAST Highlight(업계 표준) 또는 SonarQube(대체재, 단 클라우드 준비도·벤치마크 불가)
- **수동 평가**: 비즈니스 가치·데이터 결합도는 이해관계자 워크숍
- **협업 원칙**: 개발·운영·비즈니스·재무팀이 함께 평가하여 편향 최소화

### ③ 전환 적합도 판정

**건강도 → 6R 매핑 가이드**

| 건강도 점수 | 비즈니스 高 | 비즈니스 低 |
|-----------|-----------|-----------|
| 양호(16~20) | Rehost 또는 전환 보류 | Retain (현행 유지) |
| 보통(11~15) | Refactor / Replatform | Replace (SaaS 교체) |
| 취약(6~10) | Rearchitect / Rebuild | Retire (폐기) |
| 심각(4~5) | Rebuild (재구축 불가피) | Eliminate (즉시 폐기) |

**TIME 모델 (Gartner)** — 비즈니스 가치×기술 품질 2×2 매트릭스

| 범주 | 조건 | 비중 예시 | 전략 |
|------|------|---------|------|
| **T**olerate | 비즈니스 低 + 기술 高 | 15개(30%) | 당장은 유지, 중기 전환 계획 |
| **I**nvest | 비즈니스 高 + 기술 高 | 12개(24%) | Refactor/Replatform으로 클라우드 최적화 |
| **M**igrate | 비즈니스 高 + 기술 低 | 15개(30%) | Rearchitect/Rebuild 필요 |
| **E**liminate | 비즈니스 低 + 기술 低 | 8개(16%) | 폐기 또는 SaaS 교체 |

> **TIME vs 6R**: TIME은 "언제", 6R은 "어떻게" 전환할 것인가 결정. 경영진 소통에 효과적.

### 서비스 경계 식별 (Rearchitect/Rebuild 대상)

DDD 전략 설계로 마이크로서비스 경계 도출.

| 기법 | 참여자 | 산출물 |
|------|-------|-------|
| Event Storming | 현업(도메인 전문가) + 개발자 | 도메인 이벤트 맵, 바운디드 컨텍스트 경계 |
| Context Map 작성 | 아키텍트 + 개발자 | 바운디드 컨텍스트 간 관계도 |
| 도메인 용어 사전 | 현업 + 개발자 | 유비쿼터스 언어(동일 용어 = 동일 의미) |

> DDD의 바운디드 컨텍스트 = 마이크로서비스 경계의 가장 강력한 근거.  
> Rehost/Replatform/Refactor 대상은 구조 변경이 없어 서비스 경계 식별 불필요.

## 2.4 Step 3 — 전략 수립

### ① 6R 전략 상세화

**전략별 실행 방안·기간·예산·리스크**

| 전략 | 실행 내용 | 소요 기간 | 예산 규모 | 리스크 |
|------|---------|----------|----------|--------|
| Rehost | 클라우드 VM 이전, 설정 변경만 | 수주 | ★☆☆☆☆ | 낮음 |
| Replatform | OS/DB/WAS 교체(Unix→Linux, Oracle→PostgreSQL) | 수주~수개월 | ★★☆☆☆ | 낮~중 |
| Refactor | 모놀리스 유지, 클라우드 서비스 연동(S3·Redis 등) | 수개월 | ★★★☆☆ | 중간 |
| Rearchitect | DDD 기반 MSA 분리 | 6개월~1년+ | ★★★★☆ | 높음 |
| Rebuild | 새 기술 스택으로 재구축 | 1년+ | ★★★★★ | 높음 |
| Replace | 상용 SaaS로 교체(SAP, Salesforce 등) | 수개월 | ★★★☆☆(라이선스) | 중간 |

**시스템 단위 예산 산정**  
시스템별 전환 예산 = (투입 인력 × 인건비 × 소요 기간) + 도구·인프라 비용 + 예비비(15~20%)

| 전략 | 평균 투입 인력 | 평균 기간 | 예산 범위(시스템당) |
|------|------------|---------|-------------------|
| Rehost | 2~3명 | 2~4주 | 3천만~5천만 |
| Replatform | 3~5명 | 1~3개월 | 5천만~1.5억 |
| Refactor | 3~5명 | 2~4개월 | 1~2억 |
| Rearchitect | 5~10명 | 6~12개월 | 3~8억 |
| Rebuild | 5~15명 | 12개월+ | 5~15억 |
| Replace | 2~3명 + SaaS 비용 | 2~4개월 | 1~3억(+ 연간 라이선스) |

> **예비비 15~20% 필수** — Scope Creep, 예상치 못한 기술 이슈 대비.  
> 단일 숫자 금지 — 범위로 제시(약속은 리스크가 됨).

**전체 포트폴리오 일정·예산 (50개 시스템 기준 예시)**

| Phase | 기간 | 우선순위 | 대상 | 6R 전략 | 예산 |
|-------|------|---------|------|--------|------|
| Phase 0 | 1~2개월 | — | 현황분석 + 전략 수립 | — | 컨설팅 |
| Phase 1 | 1~3개월 | 1순위 Quick Win | 15개 시스템(구조 변경 없음) | Rehost/Replatform/Refactor | 소~중 |
| Phase 1' | 1~2개월 | 파일럿 | 2순위 중 2~3개 시스템 | Rearchitect(소규모 검증) | 소규모 |
| Phase 2 | 3~6개월 | 2순위 | 데이터 의존도 낮은 서비스 | Rearchitect | 중 |
| Phase 3 | 6~12개월 | 3~4순위 | 부하 변동 큰 + 핵심 차별화 | Rearchitect/Rebuild | 대 |
| Phase 4 | 지속 | — | 대체/폐기 서비스 | Replace/Retire | 라이선스 절감 |

- Phase 1(Quick Win)과 Phase 1'(파일럿)은 **병렬 실행 가능**
- Phase 1' 완료 후 L3 정량화 확정 → Phase 2 GO/NO-GO 의사결정

**파일럿 대상 선정 기준**

| 기준 | 조건 |
|------|------|
| 비즈니스 등급 | B등급(지원 업무) |
| 롤백 가능성 | 실패 시 원복 용이 |
| 데이터 결합도 | 독립 DB, 외부 의존성 적음 |
| 팀 준비도 | 전환 의지가 높은 팀 |
| 성과 가시성 | 전환 효과 빠르게 측정 가능 |

> 파일럿 목적 = **성공이 아닌 학습**. 기간 1~2개월, 산출물은 실측 데이터(L3) + Lessons Learned.

### ② TCO 분석

**As-Is TCO (현행 유지비)** — 기술 부채 비용 포함

| 비용 항목 | 세부 내용 | 산정 방법 |
|---------|---------|---------|
| 인프라 | 서버·스토리지·네트워크·IDC | 월별 청구서 합산 |
| 라이선스 | OS·DB·미들웨어·상용 SW | 연간 계약서 |
| 인건비 | 개발·운영·보안 인력 | FTE × 인건비 단가 |
| 장애 비용 | 장애 대응 인건비, 매출 손실, SLA 위약금 | 인시던트 로그 × 평균 비용 |
| 기회 비용 | 신기능 개발 지연에 따른 시장 기회 손실 | 추정(정성적) |
| 기술 부채 유지비용 | 유지보수 추가 공수 + 보안 리스크 + 인재 이탈 | 건강도 평가 결과 합산 |

> 기술 부채 비용을 As-Is에 포함하면 "전환 안 했을 때 누적 비용"이 가시화되어 BEP가 앞당겨짐.

**To-Be TCO (전환 후)**

| 비용 항목 | 세부 내용 |
|---------|---------|
| 클라우드 | IaaS/PaaS 월 비용, 데이터 전송 |
| 전환 투자 | 개발 인건비, 컨설팅, 도구 라이선스 |
| 학습 비용 | 교육·인증·파일럿, 생산성 저하 기간 |
| 병렬 운영 | 전환 기간 중 레거시 + 신규 이중 인프라 |

**BEP(손익분기점)**  
`BEP = 전환 투자 비용 ÷ (월간 As-Is TCO − 월간 To-Be TCO)`  
평균적으로 18~36개월. 3년 또는 5년 TCO 분석 권장(둘 다 제시도 좋음).

> '보이지 않는 비용'(장애 대응·보안 사고·기회 비용)을 As-Is에 포함해야 정확한 BEP.

### ③ 리스크 평가

| 리스크 영역 | 대표 리스크 | 완화 전략 |
|-----------|-----------|---------|
| 기술 | 데이터 마이그레이션 실패, 성능 저하, 호환성 이슈 | Phase 1' 파일럿으로 조기 검증, 병렬 운영 + 롤백 계획 |
| 조직 | 기술 역량 부족, 변화 저항, 핵심 인력 이탈 | 변화관리 조기 착수, Quick Win 성과로 신뢰 구축, 사전 교육 |
| 비즈니스 | 서비스 중단, 고객 이탈, 규제 미준수 | Strangler Fig + Rolling/Canary/Blue-Green 배포, 컴플라이언스 체크리스트 |
| 일정 | Scope Creep, 병렬 운영 장기화, 예산 초과 | MVP 범위 고정, Phase별 GO/NO-GO 게이트, 예비비 15~20% |

> 각 리스크에 "발생 확률 × 영향도" 매트릭스 작성, 상위 5개 리스크는 구체 완화 전략과 책임자 지정.

**거버넌스 회의체**

| 회의체 | 역할 | 참석자 | 주기 | 주요 의사결정 |
|-------|------|-------|------|------------|
| 스티어링 커미티 | 경영진 보고, GO/NO-GO | CIO/CTO, PM, EA 리드 | 월 1회 | Phase 전환 승인, 예산 추가 승인 |
| 전환 워킹그룹 | 실무 진행, 이슈 해결 | 개발·운영·비즈니스 리드 | 주 1회 | 기술 이슈 해결, 일정 조정 |
| 아키텍처 리뷰 보드 | 설계 검토, 표준 준수 | EA, 아키텍트 | 스프린트/Phase별 | 서비스 경계, API 설계 승인 |

**Phase별 가드레일**

| Phase | 파이프라인 보안·품질 | 운영 테스트 | 변화관리 |
|-------|----------------|---------|---------|
| Phase 1 (Quick Win) | SAST + SCA + 회귀 + DAST 파이프라인 구축 | — | Quick Win 성과 → 신뢰 확보 |
| Phase 1' (파일럿) | + E2E 테스트 | — | 파일럿 팀 교육, Lessons Learned |
| Phase 2 (데이터 의존도 낮은 Rearchitect) | + Contract Testing | — | 전환 대상 팀 확대 교육 |
| Phase 3 (부하 큰 + 핵심 서비스) | + 보안 감사 자동화 | Chaos Engineering 정기 실행 | 전사 DevOps 문화 정착 |
| Phase 4 (Replace/Retire/폐기) | 파이프라인 지속 운영 | Chaos Eng. 지속 | 지속적 학습 문화 |

**Phase 전환 게이트(GO/NO-GO)**

| 게이트 | 시점 | GO 조건 | NO-GO 시 액션 |
|-------|------|--------|-------------|
| Gate 1 | Phase 1'(파일럿) 완료 후 | L3 정량화 확정, 파일럿 성공 기준 달성 | 파일럿 연장 또는 전략 재검토 |
| Gate 2 | Phase 2 완료 후 | 2순위 전환 완료, Contract Testing 통과 | Scope 축소 또는 일정 조정 |
| Gate 3 | Phase 3 완료 후 | 핵심 서비스 전환 완료, Chaos Engineering 통과 | 안정화 기간 연장 |

**테스트 전략 ① — CI/CD 파이프라인 통합(DevSecOps)**

`코드 커밋 → 앱 빌드 → 단위 테스트 → SAST → 컨테이너 이미지 빌드 → SCA → 회귀 테스트 → Contract Testing → 스테이징 → DAST → 운영`

| 구분 | 테스트 | 설명 | 실패 시 | 도구 |
|------|-------|------|--------|------|
| 보안 | SAST | 소스 정적 분석 보안 취약점 탐지 | 품질 게이트 미달 시 경고/배포 차단 | SonarQube, Checkmarx |
| 보안 | SCA | OS·라이브러리·컨테이너 이미지 취약점 스캔 | 배포 차단 | Trivy, Snyk, Grype |
| 품질 | 회귀 테스트 | 코드 변경 후 기존 기능 정상 동작 확인 | 배포 차단 | Selenium, JUnit, pytest |
| MSA | Contract Testing | 서비스 간 API 계약 자동 검증(MSA 성숙도 필요) | 배포 차단 | Pact, Spring Cloud Contract |
| 보안 | DAST | 실행 중인 앱 런타임 취약점 탐지 | 운영 배포 차단 | OWASP ZAP, Burp Suite |

> **Shift-Left**: 보안·품질 검증을 운영이 아닌 개발·빌드 단계로 앞당겨 조기 발견.

**테스트 전략 ② — 운영 환경 복원력 테스트**

- **Chaos Engineering**: 운영/스테이징 환경에 의도적 장애 주입, 시스템 복원력 검증
- **적용 시점**: Phase 3~
- **도구**: Netflix Chaos Monkey, Gremlin, Chaos Mesh
- **주기**: 주 1회 정기 실행
- **특징**: CI/CD 파이프라인 외부 — 별도의 운영 품질 검증 활동. 핵심 서비스 전환 시 필수.

| 2 | 2순위 전환 팀 | MSA 설계(DDD), DevSecOps, Contract Testing | 실무 교육 + 파일럿 팀 코칭 |

| 4 | 전사/운영팀 | FinOps, SRE | 지속 교육 |

**조직 구조 + 문화 전환**

| 영역 | As-Is | To-Be | 전환 시점 |
|------|-------|-------|---------|
| 조직 구조 | 기능 조직(개발/운영 분리) | 제품 중심 소규모 다기능 조직 | Phase 2~3 시범 → Phase 4 확대 |
| 업무 방식 | 순차적(워터폴), 계획 중요·수동 배포 | 순환적(Iterative), 실험/학습, 스크럼/칸반, CI/CD 자동 배포 | Phase 1부터 점진 전환 |
| 문화 | 수직적/폐쇄적, 실패 부정적 | 수평적/개방적, 실패 긍정적, 변화 적응 | Phase 1: "실패 허용" 선언 → Phase 3: 정착 |
| 성과 측정 | 프로젝트 완료 기준 | KPI 기반 지속 측정(DORA Metrics) | Phase 1'부터 측정 시작 |

> "기술을 바꾸는 건 6개월, **사람을 바꾸는 건 2년**" — 변화관리는 기술 전환보다 먼저, 오래 지속.  
> AM 프로젝트 실패의 1위 원인 = 기술이 아닌 조직의 변화.  
> 기술 전환(WHAT)만 하고 일하는 방식(HOW)과 마인드셋(WHY)을 바꾸지 않으면 반쪽 전환.

## 요약

AM 전환 여정은 **WHY → 현황분석 → 전략수립 → 실행 → 운영최적화** 5단계.  
정량화는 L1(벤치마크) → L2(자동 측정) → L3(파일럿) 3단계로 부담 없이 접근.  
변화관리는 Step 1부터 **조기 착수**, 전 구간 관통 테마로 운영.
