# 6R 전략 정의

> AM 커리큘럼 기준 6R: **Rehost · Replatform · Refactor · Rearchitect · Rebuild · Replace**.
> 왼쪽 → 오른쪽으로 갈수록 복잡도·비용·장기 비즈니스 가치 증가.
>
> 6R는 "어떻게 전환하는가"의 6가지 전환 전략. **Retain/Retire** 는 6R 외 별도 결정 카테고리
> (시스템에 6R 자체를 적용할지 여부의 결정).
>
> 본 정의는 [`references/am/01-am-overview.md`](../am/01-am-overview.md) 와 일치하며,
> [`references/am/02-analysis-and-strategy.md`](../am/02-analysis-and-strategy.md) 의 건강도→6R 매핑과 정합.

## 6R 한눈에 보기

### Quick Win 군 (구조 변경 없음)

| # | 전략 | 한 줄 정의 | 비용 | 리스크 | 기간 |
|---|------|-----------|-----|--------|------|
| 1 | **Rehost** | 코드 변경 없이 이전 (Lift & Shift) | ★☆☆☆☆ | 낮음 | 수주 |
| 2 | **Replatform** | OS/DB/WAS 등 일부 최적화 후 이전 | ★★☆☆☆ | 낮음~중 | 수주~수개월 |
| 3 | **Refactor** | 모놀리스 **유지** + 코드 수정으로 클라우드 서비스 연동 | ★★★☆☆ | 중간 | 수개월 |

> Refactor는 모놀리스 구조를 유지하므로 Quick Win 군에 포함.
> 단, 코드 수정이 수반되어 Rehost/Replatform보다는 비용·기간·리스크가 큼.
> [`references/am/02-analysis-and-strategy.md`](../am/02-analysis-and-strategy.md) Phase 1 분류 기준 정합.

### AM 군 (구조적 변화 수반)

| # | 전략 | 한 줄 정의 | 비용 | 리스크 | 기간 |
|---|------|-----------|-----|--------|------|
| 4 | **Rearchitect** | 모놀리스 → 마이크로서비스 전환 | ★★★★☆ | 높음 | 6개월~1년 |
| 5 | **Rebuild** | 클라우드 네이티브로 완전 재개발 | ★★★★★ | 매우 높음 | 1년~2년+ |
| 6 | **Replace** | SaaS 또는 상용 솔루션 대체 | ★★★☆☆ (+ 라이선스) | 중간 | 수주~수개월 |

> **암기법**: 하(H)늘로 풍(P)선을 플(F)라이, 아(A)에 버(B)리고 플(P)레이
> ([`references/am/01-am-overview.md`](../am/01-am-overview.md))

### Beyond 6R — 결정 카테고리 (전환 자체를 적용하지 않는 경우)

| 카테고리 | 한 줄 정의 | 비고 |
|---------|----------|------|
| **Retain** | 현행 유지 — 차기 재평가 시점 명시 | 의존성 대기 또는 가치 유지 시 |
| **Retire** | 폐기·기능 통합 후 제거 | 사용자 적음 + 가치 ↓ |
| **Eliminate** | 즉시 폐기 (Retire의 가장 강한 형태) | 건강도 심각(4~5) + 비즈니스 ↓ |

> Retain/Retire/Eliminate는 6R 결정 이전 단계에서 평가. 예: "이 시스템에 6R 자체를 적용할 것인가?"

## 1. Rehost (Lift & Shift)

### 정의
시스템을 코드/아키텍처 변경 없이 새 인프라로 그대로 이동.

### 실행 방식
- VM 이미지 마이그레이션 도구 (AWS MGN, Azure Migrate, GCP Migrate for Compute Engine)
- 데이터베이스는 동일 엔진 유지 (예: Oracle on-prem → Oracle on EC2)
- 네트워크·방화벽·인증 등 주변 설정만 신 인프라 환경에 재구성

### 결과
- **단기 효과**: 데이터센터 비용 절감, 인프라 현대화
- **장기 한계**: 클라우드 네이티브 이점 미활용, 기술 부채 그대로 유지
- **AI ROI 영향**: 거의 없음 (시스템 진화 없음)

### 적합한 시스템
- 클라우드 이전이 시급한 시스템 (DC 임대 종료, 라이선스 만료 등)
- 기능 변경이 거의 없는 안정적 시스템
- 단기 PoC · 임시 해결책

### 부적합한 시스템
- 자주 변경·확장되는 시스템 (클라우드 네이티브 이점 미활용)
- 라이선스 비용이 큰 상용 SW (클라우드에서도 그대로 부담)

### 일반적 비용·기간 (am 커리큘럼 기준)
- **인력**: 2~3명
- **기간**: 2~4주
- **예산**: 3천만 ~ 5천만 KRW (시스템당)
- **예비비**: 10~15%
- **클라우드 활용도**: 낮음 (IaaS)

## 2. Replatform (Lift, Tinker & Shift)

### 정의
시스템의 핵심 아키텍처는 유지하되, OS/DB/WAS 등 일부 컴포넌트를 클라우드 관리형 서비스로 교체.

### 실행 방식
- DB → Managed DB (RDS, Cloud SQL, Azure SQL Managed Instance)
- 메시징 → Managed Queue (SQS, Pub/Sub, Service Bus)
- WAS → Managed Container/PaaS (ECS, App Engine, App Service)
- 상용 → 오픈소스 전환 (예: Unix→Linux, Oracle→PostgreSQL)
- CI/CD 파이프라인 표준화, 관측성(Observability) 통합

### 결과
- **단기**: 운영 부담 ↓, 자동 확장·복구 가능
- **중기**: 가동률·보안 패치·백업 자동화로 인적 비용 감소
- **AI ROI 영향**: ↑ (표준화된 환경 + 데이터 접근성 일부 개선)

### 적합한 시스템
- 비즈니스 가치 ↑, 기술 품질 △ (개선 여지 큼)
- 핵심 도메인 로직은 안정적이나 인프라 운영 부담 큰 시스템
- DB·메시징·WAS 라이선스가 비싼 상용 솔루션

### 부적합한 시스템
- 도메인 모델 자체가 노후화된 시스템 (Rearchitect/Rebuild 필요)
- 코드 결합도가 너무 높아 관리형 서비스로 분리 어려운 시스템

### 일반적 비용·기간 (am 커리큘럼 기준)
- **인력**: 3~5명
- **기간**: 1~3개월
- **예산**: 5천만 ~ 1.5억 KRW (시스템당)
- **예비비**: 15~20%
- **클라우드 활용도**: 중간 (PaaS 일부)

---

## 3. Refactor (모놀리스 유지 + 클라우드 서비스 연동)

### 정의
**모놀리스 아키텍처는 유지**하면서 코드를 수정하여 클라우드 관리형 서비스를 적극 활용.

### 실행 방식
- 캐시 → ElastiCache/Memorystore/Redis Cloud
- 파일 저장소 → S3/Cloud Storage/Blob Storage
- 인증 → Cognito/Identity Platform/Entra ID
- 메시징 → SNS/Pub/Sub/Event Hubs (애플리케이션 단에서 활용)
- 로깅·모니터링 → CloudWatch/Cloud Monitoring/Application Insights

### 결과
- **단기**: 탄력성·확장성 일부 확보 (관리형 서비스 효과)
- **중기**: 비용·운영 효율 개선
- **장기**: 모놀리스 한계는 그대로 (배포 주기·독립 확장 제한)
- **AI ROI 영향**: 중간 (관리형 서비스로 데이터 접근성 개선)

### 적합한 시스템
- 모놀리스이지만 비교적 잘 짜인 코드 + 핵심 도메인 안정
- 클라우드 네이티브 이점 일부만 필요한 시스템
- 큰 구조 변화가 위험하거나 비용 효과적이지 않은 시스템

### 부적합한 시스템
- 모놀리스 한계가 심각한 시스템 (Rearchitect 필요)
- 라이프사이클 종료가 가까운 시스템 (Rehost 충분)

### 일반적 비용·기간 (am 커리큘럼 기준)
- **인력**: 3~5명
- **기간**: 2~4개월
- **예산**: 1억 ~ 2억 KRW (시스템당)
- **예비비**: 15~20%
- **클라우드 활용도**: 높음 (PaaS/서버리스)

## 4. Rearchitect (모놀리스 → MSA)

### 정의
도메인 주도 설계(DDD) 기반으로 모놀리스를 마이크로서비스로 분해.

### 실행 방식
- Bounded Context 식별 → 마이크로서비스 경계 도출
- API 게이트웨이 + 서비스 메시 + 컨테이너 오케스트레이션 (Kubernetes)
- 데이터베이스 분리 (서비스별 DB), CQRS·이벤트 소싱 검토
- DevOps·CI/CD·관측성·SRE 실천 도입
- Strangler Fig 패턴으로 점진적 분해

### 결과
- **장기 효과**: 빠른 배포, 독립 확장, 장애 격리, 신기능 출시 속도
- **단기 비용**: 큰 투자 + 긴 기간 + 도메인 분해 학습
- **AI ROI 영향**: 높음 (DORA AI Capabilities 7개 역량과 정렬 가능)

### 적합한 시스템
- 비즈니스 가치 ↑↑ (전략 시스템) + 기술 부채 ↑
- 빠른 진화·확장이 필요한 시스템 (B2C 핵심 서비스, 디지털 채널)
- 도메인 경계가 명확하거나 명확화 가능한 시스템
- 5년 이상 운영 예정인 핵심 시스템

### 부적합한 시스템
- 라이프사이클 종료가 가까운 시스템 (Retire 후보)
- 도메인 전문가가 사실상 없는 시스템 (Rearchitect 위험 ↑↑)
- 비즈니스 가치는 있지만 변경 빈도가 매우 낮은 시스템 (Replatform 충분)

### 일반적 비용·기간 (am 커리큘럼 기준)
- **인력**: 5~10명
- **기간**: 6개월~1년
- **예산**: 3억 ~ 8억 KRW (시스템당)
- **예비비**: 20~25%
- **클라우드 활용도**: 매우 높음 (MSA + K8s + Serverless 혼합)

## 5. Rebuild (완전 재개발)

### 정의
기존 시스템을 폐기하고 새 기술 스택으로 클라우드 네이티브 시스템을 처음부터 재개발.

### 실행 방식
- As-Is 시스템은 유지보수 동결 (변화 없음)
- 새 기술 스택 선정 (모던 언어·프레임워크·클라우드 네이티브 패턴)
- 도메인 재정의 + UI/UX 재설계 가능
- 데이터 마이그레이션 + 병렬 운영 + 컷오버
- 기존 시스템 단계적 폐기

### 결과
- **장기 효과**: 가장 큰 변화 가능. 기술 부채 완전 청산. 비즈니스 모델 재설계 가능
- **단기 비용**: 가장 큰 투자 + 가장 긴 기간 + 가장 큰 리스크
- **AI ROI 영향**: 매우 높음 (처음부터 AI Capabilities 7개 모두 반영 가능)

### 적합한 시스템
- 기술 부채가 심각하여 점진적 개선 불가 (Rearchitect도 비효율)
- UI/UX 또는 비즈니스 모델 자체를 새로 정의해야 하는 시스템
- 사용 중인 언어·플랫폼이 EOL이며 마이그레이션 도구도 부재
- 핵심 차별화 가치 ↑ + 5년 이상 운영 예정

### 부적합한 시스템
- 도메인 전문가 부재 (As-Is 분석조차 어려움)
- 단기간 내 결과 필요 (Rebuild는 1년+)
- 비즈니스 가치 △ (Rearchitect 또는 Replace로 충분)

### 일반적 비용·기간 (am 커리큘럼 기준)
- **인력**: 5~15명
- **기간**: 12개월 이상 (통상 1~2년)
- **예산**: 5억 ~ 15억 KRW (시스템당, 대규모는 +)
- **예비비**: 25~30%
- **클라우드 활용도**: 매우 높음

## 6. Replace (SaaS·상용 솔루션 대체)

### 정의
자체 개발 시스템을 상용 SaaS 또는 패키지 솔루션으로 대체. (AWS 6R의 "Repurchase" 와 동일 개념)

### 실행 방식
- 솔루션 선정 (RFP, PoC, TCO 비교)
- 데이터 마이그레이션 + 통합 인터페이스 구축
- 비즈니스 프로세스 표준화 (솔루션 표준에 맞춤)
- 사용자 교육 + 변화관리
- 기존 시스템 단계적 폐기

### 대표 사례
- ERP: SAP, Oracle ERP Cloud, Workday
- CRM: Salesforce, HubSpot, MS Dynamics
- HR: Workday, Successfactors
- 그룹웨어·협업: M365, Google Workspace
- 회계·재무: 더존, 영림원, NetSuite

### 결과
- 운영·유지보수 부담 외주화
- 벤더 종속(Vendor Lock-in) 발생
- 커스터마이징 한계 → 비즈니스 프로세스 표준화 필요
- **AI ROI 영향**: SaaS 자체의 AI 역량에 의존 (조직이 직접 영향 X)

### 적합한 시스템
- 표준화 가능한 백오피스 (HR, ERP, CRM, 회계, 그룹웨어)
- 자체 개발 차별화 가치가 낮은 시스템
- 자체 운영 비용 > SaaS 구독료 (3~5년 TCO 비교 후)

### 부적합한 시스템
- 비즈니스 차별화 핵심 시스템
- 데이터 주권·컴플라이언스 제약으로 외부 SaaS 사용 불가
- 기존 시스템과 깊게 통합되어 있어 분리 비용이 SaaS 비용 초과

### 일반적 비용·기간 (am 커리큘럼 기준)
- **인력**: 2~3명 + SaaS 비용
- **기간**: 2~4개월 (대형 ERP는 6개월+)
- **예산**: 1억 ~ 3억 KRW (일회성) + 연간 라이선스
- **예비비**: 15~25%
- **클라우드 활용도**: 높음 (SaaS)

---

## Beyond 6R — Retain (현행 유지)

### 정의
현 시점에서는 변경 없이 유지. 차기 검토 시점 명시.

### 실행 방식
- 모니터링·운영 지속, 기능 변경 동결
- 차기 재평가 시점 등록 (예: 6개월 후, 1년 후)
- 운영 비용·기술 부채 추세 추적

### 결과
- 단기 비용·리스크 회피
- 기술 부채 누적 가능성

### 적합한 시스템
- 라이프사이클 종료가 가까워 곧 Retire 예정
- 다른 의존 시스템의 변경을 기다려야 하는 시스템 (의존성)
- 현 단계에서 ROI가 명확하지 않아 다음 Phase에서 재평가할 시스템

### 부적합한 시스템
- 보안 취약점 누적 시스템 (즉시 조치 필요)
- 라이선스·SaaS 만료 임박

---

## Beyond 6R — Retire (폐기)

### 정의
시스템을 폐기. 기능은 통합되거나 단순히 제거.

### 실행 방식
- 사용자 통보 + 마이그레이션 경로 제공
- 데이터 보존 정책에 따른 아카이빙
- 의존 시스템의 인터페이스 제거
- 라이선스 해지, 인프라 회수

### 결과
- 운영비 즉시 절감
- 기술 부채·보안 위험·인력 의존 해소

### 적합한 시스템
- 사용자 0~극소수 + 비즈니스 가치 ↓
- 다른 시스템에서 동일 기능을 더 잘 제공
- 규제·법적 보존 요건만 충족하면 즉시 제거 가능

### 일반적 비용·기간
- **인력**: 0.5~1명 (영향 분석) + 데이터 엔지니어 1~2명
- **기간**: 1~3개월 (데이터 마이그레이션·아카이빙·통보)
- **예산**: 일회성 (이후 비용 0)
- **예비비**: 10%

---

## Beyond 6R — Eliminate (즉시 폐기)

### 정의
Retire의 가장 강한 형태. 건강도가 심각(4~5점)이고 비즈니스 가치가 낮아
재평가 없이 즉시 폐기 결정 가능한 시스템.

### 실행 방식
- 영향 분석 워크숍 단축 (1주 이내)
- 즉각적 폐기 결정 → 사용자 통보 → 데이터 아카이빙

### 적합한 시스템
- 건강도 심각(4~5) + 비즈니스 가치 1~2
- 보안·컴플라이언스 위반 임박 + 즉시 대체 가능
- 사용자 거의 없음 + 운영비 누적

> 본 카테고리는 [`references/am/02-analysis-and-strategy.md`](../am/02-analysis-and-strategy.md)
> 의 "건강도 → 6R 매핑" 표 (심각 4~5 + 비즈니스 低 → Eliminate) 과 일치.
