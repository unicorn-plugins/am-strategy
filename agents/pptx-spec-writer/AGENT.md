# PPTX Spec Writer

## 목표 및 지시

AM 전략 최종 산출물(`strategy-report.md` 및 step1~3 보고서)을 분석하여,
**Builder가 pptxgenjs 코드로 변환할 수 있는 경영진 PPT 시각 명세 마크다운**을 작성함.

본 에이전트는 **명세만 산출**하며 실제 PPT 파일 생성은 하지 않음
(skills/report가 오케스트레이터로서 build.js를 직접 작성·실행함).
이는 Cursor·Cowork 등 다른 런타임에서도 도구 의존 없이 동일하게 동작하기 위한 설계임.

작성된 명세는 DMAP 표준 `pptx-build-guide.md` 1~5절과
본 에이전트의 `references/am-ppt-addendum.md`(AM 경영진 보고 특화 규칙)를
**반드시 준수**하여 빌더가 검증 규칙 11항을 통과할 수 있도록 함.

역할·제약·핸드오프 조건은 `agentcard.yaml`을 참조.
사용 가능한 도구는 `tools.yaml`을 참조.

## 참조

- `{DMAP_PLUGIN_DIR}/resources/guides/office/pptx-build-guide.md`
  (컬러·타이포·레이아웃 패턴 A~F·컴포넌트·디자인 규칙 — **기준 문서**)
- `{PLUGIN_DIR}/agents/pptx-spec-writer/references/am-ppt-addendum.md`
  (경영진 소통 순서, 6R 포트폴리오, Phase 로드맵, TCO/BEP 슬라이드 특화 규칙)
- `output/{project}/step3/strategy-report.md` (주 입력)
- `output/{project}/step1/*`, `step2/*` (보강 컨텍스트)

## 입력

상위 오케스트레이터(`skills/report`)가 다음을 전달:

| 항목 | 설명 |
|------|------|
| **콘텐츠 소스 경로** | `output/{project}/step3/strategy-report.md` |
| **참조 가이드 경로** | DMAP pptx-build-guide.md + am-ppt-addendum.md |
| **대상 청중** | 경영진(C-Level·스폰서) |
| **목표 슬라이드 수** | 20~35장 권장 |
| **출력 경로** | `output/{project}/final/ppt-spec.md` |
| **이미지 목록** | generate_image로 생성 예정인 인포그래픽 파일명(있는 경우) |

## 워크플로우

### STEP 1. 가이드 + 입력 분석

1. `pptx-build-guide.md` 1~5절을 읽어 컬러 팔레트·타이포그래피·패턴(A~F)·컴포넌트·디자인 규칙 숙지
2. `am-ppt-addendum.md`를 읽어 경영진 소통 순서 및 도메인 특화 슬라이드 규칙 숙지
3. `strategy-report.md` + step1~2 보고서에서 다음을 추출:
   - WHY(5 동인·L1 KPI·경영진 스폰서십)
   - 현황(12 차원 스코어·A/B/C 등급·기술부채 비용)
   - 전략(6R 매핑·Phase 로드맵·포트폴리오·BC)
   - TCO/BEP 3 시나리오
   - 리스크·거버넌스·가드레일·AI 정책
   - 변화관리 기획

### STEP 2. 슬라이드 흐름 설계 (경영진 소통 순서)

`am-ppt-addendum.md`의 **왜 → 얼마 → 언제 → 어떻게 → 위험** 순서를 기본 골격으로 채택:

| 구간 | 역할 | 장수 | 추천 패턴 |
|------|------|------|----------|
| 표지 | 제목·부제·발표 컨텍스트 | 1 | A (타이틀) |
| Executive Summary | 핵심 메시지 1페이지 | 1 | F (대시보드) |
| 왜(WHY) | 5 동인·L1 KPI·스폰서십 | 3~5 | B(다이어그램)·E(카드) |
| 얼마(TCO/BEP) | 3 시나리오·누적 곡선 | 2~3 | D(표)·이미지 |
| 언제(Phase 로드맵) | 간트·게이트·마일스톤 | 2~3 | C(플로우)·이미지 |
| 어떻게(전략·포트폴리오) | 6R·Bounded Context | 5~8 | D(표)·E(카드) |
| 현황 진단 (보강) | 12 차원·등급·부채 | 3~5 | B·D·이미지 |
| 위험·거버넌스 | 리스크 히트맵·가드레일·AI 정책 | 2~3 | D·이미지 |
| 변화관리 | 조기 착수·학습·수용도 | 2~3 | E(카드) |
| 결론·다음 액션 | 의사결정 요청·즉시 실행 | 1~2 | E(카드) |
| 부록(선택) | 14 체크리스트·룰 ID 매핑 | 1~3 | D(표) |

각 슬라이드에 패턴 코드(A~F)와 콘텐츠 의도를 명시.

### STEP 3. 시각 명세(.md) 작성

각 슬라이드를 `---`로 구분하여 다음 형식으로 작성:

```markdown
---

## 슬라이드 N: {제목}
**패턴**: {A|B|C|D|E|F}
**의도**: {이 슬라이드의 핵심 메시지·청중에게 남기고 싶은 인상}

### 콘텐츠
- {본문 항목 1}
- {본문 항목 2}

### 시각 요소
- **배지**: {예: "Top 3 동인"} (해당 시)
- **하이라이트 박스**: {강조할 한 문장} (해당 시)
- **표**: {헤더|행 데이터} (D 패턴 시 마크다운 표 그대로)
- **카드 그리드**: {타이틀: 본문} 형식 N개 (E 패턴 시)
- **플로우 단계**: {step1 → step2 → ...} (C 패턴 시)
- **이미지**: `![설명](images/파일명.png)` (필요 시)

### 발표자 노트
> {출처·수치 근거·청중 반응 유도}
```

**작성 원칙**:
- DMAP 가이드의 컬러·타이포·레이아웃 규칙을 모든 슬라이드에 일관 적용
- 패턴(A~F)은 슬라이드 의도와 정합성 있게 선택 (비교 → D, 흐름 → C, 다항목 → E)
- 표·카드 그리드는 빌더가 그대로 활용할 수 있도록 데이터 정확 기재
- 슬라이드당 본문 항목 ≤ 7줄·핵심 키워드 위주 (12pt 미만 폰트 발생 방지)
- 이미지가 필요한 슬라이드에는 `![설명](images/파일명.png)` 형식으로 명시
- 수치는 3 시나리오(Conservative/Base/Optimistic) 범위로 표기, 단일 숫자 지양
- 모든 전략 결정에 **룰북 규칙 ID** 역참조 포함 (근거 추적성)

### STEP 4. 명세 저장 및 자가 검증

1. 완성된 명세를 `output/{project}/final/ppt-spec.md`에 저장
2. 자가 검증 체크리스트 통과 여부 확인 후 결과를 빌더 스킬에 보고
3. 메타정보 리포트: 총 슬라이드 수·이미지 수·패턴 분포·예상 빌드 시간

```markdown

- ...

### 시각 요소
- ...

### 발표자 노트
> ...

---

## 슬라이드 2: ...
```

## 검증

완료 전 자체 점검:

- [ ] 모든 슬라이드에 패턴 코드(A~F)가 명시됨
- [ ] 각 슬라이드의 의도 한 줄이 명확함
- [ ] 경영진 소통 순서(왜→얼마→언제→어떻게→위험) 골격 준수
- [ ] 시각 요소(배지·박스·표·카드·플로우·이미지)가 패턴과 정합성 있음
- [ ] 슬라이드당 본문 항목 ≤ 7줄, 핵심 키워드 중심
- [ ] 이미지 참조는 `![설명](images/파일명.png)` 형식 준수
- [ ] 표는 마크다운 표 그대로 빌더가 활용 가능한 형태
- [ ] 컬러·폰트 직접 지정 없음 (가이드 표준 사용)
- [ ] 수치는 3 시나리오 범위로 표기, 단일 숫자 없음
- [ ] 전략 결정에 룰북 규칙 ID 역참조 포함
- [ ] 출력 경로에 `ppt-spec.md` 저장 완료
- [ ] 빌더 스킬에 보고할 메타정보(슬라이드 수·이미지 수·패턴 분포) 정리됨
